# Подключение сайта ресторана к брокеру

Этот документ описывает контракт, по которому **любой** клиентский сайт ресторана
(новый, существующий, на любом стеке) подключается к Multi-Tenant Restaurant CRM API Broker.
Изоляция между ресторанами обеспечивается ключом `X-Restaurant-Key` — он однозначно
определяет `restaurant_id` (tenant) на стороне брокера, и все данные (столы, заказы,
бронирования, меню) автоматически скоупятся в его рамках.

Референсная реализация — `vitemotion-dark-react` (`src/config.ts`, `src/broker.ts`,
`CheckoutModal.tsx`, `ContactSection.tsx`). Любой другой сайт повторяет тот же контракт.

## 0. Два способа подключения

Сайту ресторана не нужно реализовывать бронь и заказ самому — есть два варианта:

1. **Кнопка-переход в универсальный клиентский портал брокера** (`/portal/<api_key>`,
   см. раздел 2.1) — один общий React-фронтенд на брокере обслуживает бронь столов и
   заказ еды для **любого** тенанта; конкретный ресторан определяется ключом в URL.
   Так сейчас сделано в `ContactSection.tsx` — там больше нет формы бронирования,
   только кнопка с ссылкой на портал (`CLIENT_PORTAL_URL` из `src/config.ts`).
2. **Прямая интеграция с `/client/*` API** (разделы 3+) — собственный UI сайта сам
   вызывает эндпоинты брокера. Так у `vitemotion-dark-react` до сих пор работает
   меню/корзина/чекаут (`CheckoutModal.tsx` → `broker.ts`) — это решение клиента,
   независимое от выбора по брони.

Оба способа используют один и тот же `X-Restaurant-Key` и одинаково изолированы по тенанту.

## 1. Получение ключа тенанта

Каждый ресторан — это `Restaurant` со своим `id` и `api_key`. Ключ выдаётся:

- через публичную регистрацию `POST /api/v1/auth/register` — основной путь для новых
  клиентов SaaS: по одноразовому коду приглашения (раздел 1.1) создаётся независимый
  tenant + аккаунт основателя (`founder`) со свежим `api_key`;
- вручную через сид (`server/db.ts → seed()`), как сделано для демо-тенанта `rest_tenant_c`
  (`api_key_tenant_c_2026`) — специально предназначен для интеграции внешних демо-сайтов;
- или через `POST /api/v1/crm/restaurants` (только `super_admin`), что регистрирует новый
  ресторан в SaaS и возвращает свежий `api_key`.

Ключ передаётся сайтом ресторана в каждом клиентском запросе заголовком:

```
X-Restaurant-Key: <api_key>
```

### 1.1 Регистрация и роли персонала

Публичная регистрация создаёт **только** аккаунт основателя — это защита от того,
чтобы CRM подключали произвольные посторонние (`founder_invite_codes` выдаёт лично
`super_admin` каждому реальному клиенту):

```
POST /api/v1/auth/register
{ "restaurant_name": "...", "email": "...", "password": "...", "invite_code": "..." }
```

Код одноразовый (`used_at` проставляется сразу). Ответ содержит `token` и `user`
с массивом `restaurants: [{ id, name, api_key }]` — это первый ресторан основателя.

Дальше **внутри CRM** основатель сам заводит остальной персонал через
`POST /api/v1/crm/employees` с ролью `manager | hostess | chef` (founder и
super_admin публично не создаются). Роли и их видимость:

| Роль          | Доступ                                                              |
|---------------|----------------------------------------------------------------------|
| `founder`     | Все свои рестораны, финансы, сотрудники, переключение между ними     |
| `manager`     | Всё то же, что у founder, но заперт на один (текущий) ресторан       |
| `hostess`     | Только бронирования и карта столов                                   |
| `chef`        | Только заказы и кухонный экран                                       |
| `super_admin` | Глобальный реестр всех тенантов (служебная роль поставщика CRM)      |

Основатель, владеющий несколькими ресторанами, переключает "активный" тенант через
`POST /api/v1/crm/founder/switch-restaurant` (переиздаёт JWT с новым `restaurant_id`),
добавляет новые через `POST /api/v1/crm/founder/restaurants` (каждый — отдельный tenant
со свежим `api_key`) и архивирует через `DELETE /api/v1/crm/founder/restaurants/:id`
(блокируется при незавершённых заказах или будущих бронированиях).

## 2. Базовый URL и переменные окружения

```
VITE_BROKER_API_BASE=http://localhost:3000/api/v1               # в проде — реальный домен брокера
VITE_RESTAURANT_KEY=api_key_tenant_c_2026                        # ключ конкретного тенанта
VITE_CLIENT_PORTAL_URL=http://localhost:3000/portal/api_key_tenant_c_2026  # опционально, см. 2.1
```

CORS у брокера настроен permissive (отражает любой `Origin`, разрешает заголовки
`Content-Type`, `Authorization`, `X-Restaurant-Key`) — кросс-доменные запросы с сайта
ресторана работают без дополнительной настройки.

### 2.1 Универсальный клиентский портал (`/portal/<api_key>`)

Брокер сам раздаёт единый клиентский фронтенд (`src/ClientPortal.tsx`, маршрутизация —
`src/main.tsx` по `window.location.pathname`, без отдельного сервер-роута: и dev-режим
Vite, и прод-фоллбэк `app.get("*", ...)` в `server.ts` уже отдают `index.html` на
любой неизвестный путь). Открыв `<домен_брокера>/portal/<api_key>`, гость попадает на
страницу с двумя вкладками — бронь стола (карта зала) и заказ еды (меню + корзина +
доставка/самовывоз/в зале + симуляция оплаты) — без аккаунта и без входа в CRM.

Один и тот же бандл обслуживает любой тенант: какой ресторан показывать, определяется
исключительно ключом из URL, а не сборкой/деплоем под конкретного клиента. Если
`VITE_CLIENT_PORTAL_URL` не задана, `src/config.ts` сайта ресторана собирает её сам —
origin из `VITE_BROKER_API_BASE` + `/portal/` + `VITE_RESTAURANT_KEY`.

## 3. Эндпоинты для клиентского сайта (`/api/v1/client/*`)

Это тот же контракт, на котором работает и универсальный портал (раздел 2.1), и
прямая интеграция произвольного сайта (раздел 0, способ 2). Все требуют
`X-Restaurant-Key`, кроме `payments/webhook` (он привязывается к заказу по `order_id`,
который уже несёт tenant-контекст).

### `GET /client/restaurant`
Возвращает `{ id, name, menu }` — публичную карточку ресторана и доступное меню.

### `GET /client/tables`
Возвращает `{ tables: [{ id, table_number, capacity, x_pos, y_pos, current_status }] }`.
`current_status` — `free | reserved | occupied`. Сайт использует это для подбора
свободного стола под бронирование/предзаказ — **никогда не присылайте случайный
`table_id`**, он валидируется на принадлежность тенанту.

### `POST /client/orders`
Создаёт заказ. Тело запроса:

```jsonc
{
  "total_amount": 4900,
  "items": [{ "dish_name": "...", "quantity": 1, "price_per_unit": 4900 }],
  "delivery_type": "in_restaurant" | "takeaway" | "delivery",
  "table_id": "tbl_...",          // обязателен для in_restaurant
  "delivery_address": "...",      // обязателен для delivery
  "customer_name": "...",         // обязателен для delivery (контакт для курьера)
  "customer_phone": "..."         // обязателен для delivery
}
```

Бизнес-правила, которые проверяет брокер (400/404 при нарушении):
- `in_restaurant` без `table_id` — отказ;
- `delivery` без `delivery_address` или без `customer_name`/`customer_phone` — отказ;
- `table_id`, принадлежащий другому тенанту или несуществующий — отказ (защита от спуфинга).

SLA на готовность, который проставляется автоматически: `in_restaurant` — 15 мин,
`takeaway` — 25 мин, `delivery` — 45 мин.

Ответ: `{ message, order_id, order, items }`, `payment_status: "pending"`.

### `POST /client/payments/webhook`
Подтверждение оплаты от платёжного провайдера (или, в демо без реального шлюза, —
немедленный вызов сразу после создания заказа). Тело:

```json
{ "order_id": "ord_...", "idemp_key": "уникальный_токен", "status": "success", "amount": 4900 }
```

`idemp_key` — обязательная защита от повторного списания: повторный вызов с тем же
ключом вернёт `duplicate: true` без повторной обработки. При `status: "success"`
заказ переходит в `payment_status: "paid"`, `order_status: "new"` и появляется на
кухонном экране CRM.

### `POST /client/reservations`
Бронирование стола. Тело:

```json
{ "customer_name": "...", "customer_phone": "...", "date": "YYYY-MM-DD", "time": "HH:MM", "guests_count": 4, "table_id": "tbl_..." }
```

Брокер сам проверяет овербукинг: на одном столе нельзя иметь две брони с разницей
менее 120 минут — конфликт вернётся как `409` с полями `conflicting_time`,
`table_number`. Сайт ресторана должен подбирать `table_id` сам (через `GET
/client/tables`, по вместимости и статусу `free`) — брокер не выбирает стол за вас.

## 4. Жизненный цикл заказа (state machine)

```
new → cooking → ready ┬─(in_restaurant / takeaway)──────────────→ delivered
                       └─(delivery)→ out_for_delivery ───────────→ delivered
```

`out_for_delivery` доступен только заказам с `delivery_type: "delivery"` — переход
для `in_restaurant`/`takeaway` отклоняется брокером с `400` (см.
`crmUpdateOrderStatus` в `order.controller.ts`). Эти переходы выполняет персонал
через `PATCH /api/v1/crm/orders/:id` — сайт ресторана в них не участвует.

## 5. Изоляция и ошибки

- Неверный/чужой `X-Restaurant-Key` → `401`.
- Доступ к `table_id`/ресурсу другого тенанта → `404` с понятным текстом
  ("Spoofing attempt...") — таблица/ресурс просто не находится в скоупе тенанта.
- Формат ошибок везде единообразный: `{ "error": "..." }`.
- Ошибка в обработке одного заказа/брони не валит процесс — каждый роут обёрнут в
  `asyncHandler`, исключения не приводят к падению сервера.

## 6. Что проверено end-to-end (см. также раздел тестов в репозитории)

- Создание заказов всех трёх `delivery_type`, валидация обязательных полей.
- Идемпотентность вебхука оплаты (повторный вызов не списывает второй раз).
- Полный переход статусов `new → cooking → ready → out_for_delivery → delivered`
  и блокировка `out_for_delivery` для не-`delivery` заказов.
- Овербукинг-блокировка брони (±2ч на одном столе) и защита от спуфинга
  `table_id` чужого тенанта.
- Межтенантная изоляция: `X-Restaurant-Key` одного ресторана не видит и не может
  адресовать сущности другого (отдельные ID-неймспейсы `tbl_ta_*` / `tbl_tc_*`).
- CORS preflight с произвольного `Origin` (порт сайта ресторана) проходит корректно.
- Регистрация по одноразовому коду приглашения создаёт независимый tenant (новый
  `api_key`) + аккаунт founder; повторное использование того же кода — `403`.
- Второй основатель, зарегистрированный отдельным кодом, получает отдельный tenant —
  его столы/брони/заказы не пересекаются с первым (проверено через раздельные
  `X-Restaurant-Key`).
- Founder с двумя ресторанами переключается между ними (`switch-restaurant`) без
  потери доступа; manager/hostess/chef жёстко заперты на свой `restaurant_id` из JWT.
- Архивация ресторана founder'ом блокируется при наличии незавершённых заказов или
  будущих бронирований (`409`).
- `/portal/<api_key>` рендерит карточку нужного ресторана и его меню/карту столов
  по ключу из URL — без какого-либо JWT или входа в CRM.
